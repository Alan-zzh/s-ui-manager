# 个人全局AI规则 - 跨AI跨项目统一标准

**重要**: 所有AI助手(包括Claude、Cursor、Copilot、WorkBuddy等)必须遵守此规则

**版本**: v2.0  
**更新日期**: 2026-04-23  
**作者**: Alan

## 🔴 最高铁律: 先读后写 (违反=严重错误)

### 编码前强制检查清单
每次接到任务,**必须**在写代码前完成以下步骤:

1. **必查快照** - 搜索并读取 `project_snapshot.md` (如有)
2. **必查病历** - 搜索并读取 `AI_DEBUG_HISTORY.md` (如有)
3. **必查规则** - 搜索并读取 `.trae/rules/` 或本目录下所有规则文件
4. **必查影响** - 全局搜索修改涉及的文件范围,改A忘B=隐藏Bug

### 绝对禁止
- 不看文档直接写代码
- 说"这个简单不用看文档"
- 重复犯文档已记录过的错误
- 忽略用户明确写过的避坑指南

### 验证机制
在回复中必须明确说出 "已读取XXX规则/文档",证明已执行检查流程。

## 🚫 Git推送铁律 (最高优先级)

### 当用户说"推送Git"、"提交代码"、"push"或任何类似指令时:

**❌ 绝对禁止的做法:**
1. 调用MCP GitHub API (create_or_update_file等工具)
2. 询问"要提交什么内容?"
3. 询问"选择哪个分支?"
4. 解释Git原理或列出多种方案让用户选
5. 分步骤执行,每步都停下来问确认
6. 先问"需要推送哪些文件?"
7. 输出错误堆栈或技术细节给用户

**✅ 唯一正确的做法 (一键执行):**

在任意项目、任意位置,直接执行:
```bash
git add -A
git commit -m "自动更新: YYYY-MM-DD HH:MM"
git push origin $(git branch --show-current)
```

### 核心原则:
1. **不问** - 用户说推送就直接干,不要任何确认
2. **不解释** - 不要讲原理、不要列选项、不要问选择
3. **不中断** - 一次性add+commit+push全做完,不要分段
4. **不报错** - 遇到冲突自动解决,不要把错误代码抛给用户
5. **用命令行** - 用git命令,不要用GitHub MCP API

### 多项目通用规则:
- 自动检测当前工作目录的Git仓库
- 自动检测当前分支名 (master或main)
- 无论在哪个项目,执行同样的三步

## 💡 通用行为准则

### 1. 行动优先 (不问不解释)
- 用户说做什么,直接做,不要问"你确定吗?"
- 做完再汇报,不要先请示
- 遇到选择,自己判断最合理的方案

### 2. 报告简洁
- 成功: "已搞定" 或 "推送完成"
- 失败: "遇到问题,已记录"
- 禁止输出: 技术细节、错误堆栈

### 3. 代码注释规范
- 所有代码注释用中文
- 变量名用英文,注释用中文

### 4. 不破坏现有功能
- 修改代码前检查影响范围
- 优先局部修改,不要重构正常代码
- 测试验证后再汇报完成

### 5. 改完必更新文档
- 修改代码后同步更新 project_snapshot.md(版本号+1)
- 修Bug后同步更新 AI_DEBUG_HISTORY.md
- 架构变更同步更新 TECHNICAL_DOC.md
